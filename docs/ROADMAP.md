# Роадмап: QR-коды на памятники

Краткий план разработки от UI-прототипа до продакшена. Бэкенд сначала — **один FastAPI-монолит** (проще тестировать локально), микросервисы — только когда появятся реальные нагрузки и команда.

---

## Фаза 0 — UI без бэкенда (2–3 недели) ← **сейчас**

**Цель:** показать продукт, проверить UX, собрать обратную связь.

| Задача | Результат |
|--------|-----------|
| Скелет React + Vite + TS + Tailwind + shadcn/ui | Запускается `npm run dev` |
| Главная + модалка покупки (заглушка оплаты) | Все поля, расчёт суммы, фейковый success |
| Логин / регистрация через `localStorage` | Тест: `test@qr-memory.ru` / `1234` |
| ЛК покупателя: мемориал, фото/видео upload (локально) | Лимиты пакета, превью, прогресс |
| Админ-панель (заглушки дашбордов) | Таблицы, Chart.js с мок-данными |
| Публичная страница мемориала `/m/:slug` | Как видит посетитель после скана QR |

**Не делать:** реальная оплата, email, S3, PostgreSQL.

## Фаза 1 — Backend-скелет + Auth (1–2 недели)

**Цель:** заменить `localStorage` на API, подготовить схему БД.

```
docker compose up  →  postgres + api + (опционально) minio локально
```

| Задача | Стек |
|--------|------|
| FastAPI + SQLModel/SQLAlchemy + Alembic | Миграции, модели User, Memorial, Order |
| JWT auth (access + refresh) | `/auth/register`, `/auth/login`, `/auth/me` |
| Роли: `buyer`, `admin` | Middleware / dependency |
| CORS, `.env`, pydantic-settings | Локальная разработка |
| OpenAPI → типы на фронте (опционально) | `openapi-typescript` |

**Тест:** регистрация и вход через API, мемориал CRUD.

## Фаза 2 — Медиа и лимиты пакетов (2 недели)

**Цель:** загрузка фото/видео в Object Storage, контроль лимитов.

| Задача | Детали |
|--------|--------|
| Presigned URL API | `POST /uploads/presign` → клиент грузит в Yandex S3 |
| Запись метаданных в PG | `media_files`: type, size, duration_sec, memorial_id |
| Клиентская компрессия фото | `browser-image-compression` → WebP |
| Лимиты: 40/80/200 фото, 0/20/60 мин видео | Проверка на API + дубль на фронте |
| Thumbnail worker (фон) | Celery/RQ или простой asyncio task: 400px thumb |
| Публичная галерея | CDN URL или signed GET на 1 час |

**Локально:** MinIO вместо Yandex Object Storage (тот же S3 API).

## Фаза 3 — Заказы, QR-коды, оплата (2–3 недели)

**Цель:** полный цикл «купил → оплатил → получил коды на почту».

| Задача | Детали |
|--------|--------|
| Модель Order + OrderItem + QrCode | Статусы: `draft` → `pending_payment` → `paid` → `fulfilled` |
| Интеграция **ЮKassa** (тестовый магазин) | Redirect + webhook `payment.succeeded` |
| Чеки 54-ФЗ | «Чеки от ЮKassa» в объекте `receipt` |
| Генерация QR-записей после оплаты | N кодов на заказ, уникальный `public_slug` |
| Email-сервис (заглушка → SMTP/SendGrid/UniSender) | Чек, реестр кодов, инструкции |
| Redirect `/r/{slug}` | 302 на `/m/{memorial_id}` или страницу «активируйте» |

**Тест:** ЮKassa sandbox, webhook через ngrok.

## Фаза 5 — Продакшен + масштабирование (ongoing)

| Задача | Детали |
|--------|--------|
| Деплой | VPS (Timeweb/Selectel) или Yandex Cloud: nginx + docker |
| SSL (Let's Encrypt) | Обязательно для ЮKassa |
| CDN Yandex Cloud | Перед Object Storage |
| Видео-транскодинг | FFmpeg worker → HLS (позже) |
| **Разделение на микросервисы** | Когда >1 разработчик или узкие места |

### Порядок выделения микросервисов (не с первого дня!)

1. `payment-service` — изоляция PCI, webhook ЮKassa  
2. `media-service` — upload, thumbs, транскодинг  
3. `notification-service` — email, SMS (смена пароля)  
4. `qr-service` — генерация, редиректы, аналитика сканов  
5. `memorial-service` — контент страниц  
6. `auth-service` — JWT, роли  

Между сервисами: **PostgreSQL** (схема на сервис) или события через **Redis/RabbitMQ**.
